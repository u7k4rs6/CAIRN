# Cairn: complexity and tradeoffs (single source of truth)

Every core operation in `cairn-vcs`, `cairn-transfer`, and the one platform-level
algorithm (permission resolution), with the exact class and method that implements
it, its time and space cost as written, the alternative that was rejected, and why.

This supersedes `docs/02_TECHNICAL_ARCHITECTURE.md` section 10, which was written
before the code existed and states the target, not the as-built behavior. Where the
two disagree, this file is correct; the architecture doc has been updated to point
here instead of repeating a table of its own.

**Rule this doc follows:** every figure below is derived from reading the cited
method, not carried over from the architecture doc's pre-implementation estimate.
Two places where that reading surfaced a bound worth stating more precisely than the
original doc did are called out explicitly (merge-base's pairwise domination check,
and negotiation's closure-computation cost) ‚Äî both are real, current, defensible
readings of the code, not hypothetical concerns.

| Operation | Implementation | Time | Space | Alternative rejected | Why this over that |
|---|---|---|---|---|---|
| **Object write** (content-address + hash) | `cairn-vcs/store/LooseObjectStore.java:put` (line 48), hashing via `object/ObjectId.java:hash` (line 43) and framing via `object/AbstractGitObject.java:serialize` | O(n): `serialize()` builds the `"<kind> <len>\0<body>"` frame (O(n)), `ObjectId.hash` runs one SHA-1 digest pass over it (O(n)); a `Files.exists` dedup check (O(1)) short-circuits if the content is already stored, then it is deflated and written (O(n)). | O(n) to hold the serialized frame in memory; O(1) additional for the streaming `DeflaterOutputStream` write. | Always pack on write (delta-encode against a chosen base immediately). | Every write lands in loose storage first because a base to delta against may not exist yet (or may not be a good one) at write time; packing is a separate, later operation (`PackWriter`) over a chosen object set, not a per-write decision. Loose-first keeps writes O(n) and simple; the dedup check alone already gives "identical content stored once" without needing pack machinery to be present. |
| **Object read: loose** | `store/LooseObjectStore.java:get` (line 70), deserialization via `object/GitObjects.java:deserialize` | O(1) to locate the sharded path (`pathFor`, line 43) + O(n) to inflate and deserialize (one linear scan for the header's NUL terminator in `GitObjects.deserialize`, then a byte-array copy of the body). | O(n): the inflated bytes are buffered whole (`ByteArrayOutputStream`) before deserializing. | Memory-map the compressed file and inflate on demand per access. | Not attempted; every loose read is a full inflate. Simpler, and loose objects are the "not yet packed" path, not the hot read path a real deployment would optimize (that is what packing exists for). |
| **Object read: packed** | `store/PackedObjectStore.java:get` (line 38), backed by `pack/PackReader.java:read` (line 36) | O(1) per `get` ‚Äî a `Map<ObjectId, GitObject>` lookup, since the *entire* pack was decoded once at construction. That one-time decode is O(pack size): `PackReader.read` makes a single forward pass, inflating each entry once and, for a `REF_DELTA` entry, applying `DeltaCodec.decode` exactly once against a base that was *already fully resolved* by an earlier iteration of the same pass (packs are written by `PackWriter` with bases always preceding their deltas). So decode cost is proportional to pack contents, not to chain depth √ó object count ‚Äî verified by reading the loop at `PackReader.java:52-71`, which resolves each entry against `resolved.get(baseId)`, never re-descending into an unresolved chain. | O(pack size): every decoded object is held in the map for the store's lifetime. | Lazy per-object resolution from a byte offset via a separate `.idx` file (real Git's approach), decoding and applying a delta chain only when that object is actually requested. | Rejected for implementation simplicity: no on-disk index format to design or maintain. The tradeoff is paying the full O(pack size) decode cost once at construction even if only a few objects from the pack are ever read, versus real Git's per-object lazy cost paid only for objects actually touched. |
| **Ref resolution** | `ref/FileRefStore.java:resolve` (line 41); listing at `:list` (line 75) | `resolve`: O(1) ‚Äî one small fixed-size file (`Files.exists` + `Files.readString` of a ~41-byte hex id). `list`: O(R), R = number of refs, since it walks the whole `refs/` file tree. | O(1) per resolve; O(R) to materialize the list. | A single index file mapping every ref to an id (avoids a filesystem `walk` for `list`). | Rejected: one-file-per-ref matches Git's own loose-ref layout, and `list` is not a hot path (called once per negotiation/advertisement, not per object). |
| **DAG walk** | `dag/RevWalk.java:history` (line 43, committer-time-ordered) and `:reachableFrom` (lines 66/70, unordered) | `history`: O(V log V + E), tighter than the loose O((V+E) log V). Confirmed from the code: `seen` (a `HashSet`, line 47) guards every `queue.add` (lines 49, 58), so each of the V commits is pushed onto the `PriorityQueue` exactly once ‚Äî the O(log V) queue cost applies only to those V pushes/pops. Each of the E parent edges is examined once purely to test `seen.add`, an O(1) hash-set operation, so edges do not carry a log factor; only vertices do. `reachableFrom`: O(V + E) ‚Äî a plain `Deque`-based walk with a `HashSet` visited-set, no ordering, so no log factor at all. | O(V): the `seen`/`visited` set and the result collection. | A single unordered `history` (drop committer-time ordering). | Not done: `log`'s contract (newest-first) needs the ordering; `reachableFrom`'s callers (`Ancestry`, `ObjectClosure`) don't need it, so they get the cheaper unordered walk instead of paying the log factor everywhere. This is itself the *documented* limitation from M1: ordering by committer time, not strict topological order (see `RevWalk`'s own Javadoc). |
| **Ancestor check** | `dag/Ancestry.java:isAncestor` (line 56) | O(1) when generation numbers alone can rule it out (`gen(of) <= gen(candidate)` ‚áí false, since an ancestor's generation is always strictly less than its descendant's ‚Äî line 60-63). Otherwise a pruned walk from `of` that never enqueues a parent whose generation has already dropped below `candidate`'s (lines 64-81): still O(V + E) worst case in an adversarial graph, but touches only the "frontier" between the two commits in the common case. | O(V) for the visited set on the walked-path. | No pruning: a plain O(V+E) walk every time. | Generation numbers turn the extremely common cases (one side already merged, or a fast-forward) into O(1) or a small pruned walk, at the one-time cost of computing and persisting `gen()` for every commit (`dag/GenerationNumbers.java:computeAndStore`, line 31: O(1) amortized per new commit, since parents normally already have a stored generation; O(V+E) the first time a pre-existing history is encountered ungenerated). |
| **Merge-base** | `dag/Ancestry.java:mergeBases` (line 91) | Fast path (either side already an ancestor of the other, lines 92-97): cost of two `isAncestor` calls above. General/criss-cross path (lines 99-120): two full `reachableFrom` walks, O(V+E) each; a `HashSet` intersection, O(size of the smaller reachable set); then a **pairwise domination filter** over the C common ancestors found (lines 108-119) that calls `isAncestor` for every ordered pair ‚Äî O(C¬≤) calls, each itself up to O(V+E) in the worst case. **This last cost is not stated in the architecture doc or in `Ancestry`'s own Javadoc**, and is worth being explicit about: for a criss-cross history with many common ancestors, finding the *lowest* ones is quadratic in how many common ancestors there are, not just linear in graph size. That said, every one of those C¬≤ calls is the *same* `isAncestor` method from the row above (line 111: `isAncestor(candidate, other)`), so each pairwise check still gets its O(1) generation-comparison fast path (line 61: `generationOf(of) <= candidateGen` ‚áí immediate `false`) whenever the two candidates' generations already settle the question ‚Äî which is the common case for common ancestors drawn from different depths of the graph. So typical filter cost is well below the C¬≤¬∑(V+E) worst case; only pairs whose generations don't rule things out fall through to a full walk. | O(V) for each reachable set. | A generation-bounded frontier search that stops once the remaining frontier's generation drops below the lowest candidate found. | Rejected (see `Ancestry`'s own Javadoc, lines 18-30): correctly ruling out a lower-generation-but-non-ancestor candidate in an adversarial graph needs stale-marking bookkeeping this implementation does not build. Generation numbers here only accelerate the two cases named above (the O(1) negative and the direct-ancestor fast path, including within the domination filter's pairwise checks); they do **not** change the asymptotic bound of the full criss-cross fallback, including the O(C¬≤) domination filter's worst case ‚Äî an honest complexity-class statement, not an overclaimed win. |
| **Three-way merge + conflict detection** | `merge/TreeMerger.java:merge` (line 47) orchestrating per-file `merge/FileMerge.java:merge` (line 30) | `TreeMerger`: flattens three trees via `merge/TreeFlattener.java:flatten` (line 30), O(total tree/subtree objects across all three, one `ObjectStore.get` per node); unions the three path sets into a `TreeSet`, O(P log P), P = distinct paths; for each path, O(1) three-way comparison unless both sides changed the same path differently, which invokes `FileMerge.merge`. `FileMerge`: two `MyersDiff` calls (base-vs-ours, base-vs-theirs), each O(N¬∑D); then a single merge-loop pass over the two edit-span lists that advances two indices (`ai`, `bi`) monotonically forward and never resets them (lines 40-92) ‚Äî so despite the nested `while` structure, total work across the *whole* call is O(spansA + spansB), linear in edit count, not quadratic in region-overlap count. | O(P) for the flattened path maps; O(N) per conflicted file's Myers diff (linear-space refinement). | Recursive/structural tree diff instead of flatten-to-flat-paths. | Rejected (M2 decision, restated here): flattening makes non-overlapping changes in different subtrees merge trivially and fits the per-path three-way decision table directly; the cost is that a whole-subtree move looks like N unrelated per-file changes (no rename detection ‚Äî a named, deliberate limitation, not an oversight). |
| **Diff** | `diff/MyersDiff.java:diff` (line 27), the named algorithm per the architecture doc's requirement to state it | O(N¬∑D): N = combined sequence length, D = edit-script length (edit distance); degrades toward O(N¬≤) when the inputs are almost entirely different. Space O(N) via the linear-space middle-snake divide-and-conquer (`:middleSnake`, line 98, and the recursive `:compute`, line 33), not the naive O(N¬≤) DP table. | O(N), per above. | Histogram or patience diff (anchor on lines unique to both sides). | Rejected as the *default*: Myers guarantees a minimal edit script and is exhaustively validated here against an 800+-case brute-force LCS oracle (`MyersDiffTest`); histogram/patience produce more human-readable diffs at the cost of not guaranteeing minimality, and are not implemented as an alternate `DiffStrategy` in this codebase (a real, not just theoretical, scope gap: `DiffStrategy` is an interface built for this swap, but only `MyersDiff` implements it). |
| **Packfile encode (delta)** | `pack/PackWriter.java:writePack` (line 49), byte-level delta via `pack/DeltaCodec.java:encode` (line 36) | For each of K objects: O(1) base-candidate lookup (`lastOfKind`, an `EnumMap`); if a same-kind prior object exists and its own `depthOf` is below the cap (line 62), one `DeltaCodec.encode` call is attempted, itself a byte-level `MyersDiff`, O(n¬∑d) where n = combined byte length of base+target and d = the byte-level edit distance. A delta is kept only if smaller than the full object (line 65), so total output is bounded by K full-object encodes in the worst case. **Two distinct bounds, worth separating rather than conflating:** `MAX_DELTA_DEPTH = 50` (line 41, checked via the `depthOf` map at line 62) caps delta-*chain length* ‚Äî how many deltas may stack before a full object must be written again ‚Äî which is a *reconstruction/read-side* bound (per the class's own Javadoc, lines 34-37: "reconstructing any single object is bounded work"). It does **not** bound the cost of any single `DeltaCodec.encode` call itself. That per-pair encode cost is the real exposure: two large, dissimilar same-kind objects (e.g. two big unrelated blobs that happen to be adjacent in `objectIds` and share no prior base) still cost a full byte-level Myers comparison with d approaching n, i.e. **O(n¬≤)** for that one pair, regardless of chain depth ‚Äî the chain cap bounds how many times reconstruction repeats this cost, not how expensive any one encode attempt can be. | O(pack size) being assembled in one `ByteArrayOutputStream` (line 50), plus O(largest single object body) transient during each iteration. | A rolling-hash greedy matcher (Git's real approach), which stays linear in target size even when base and target are dissimilar. | Rejected (see `DeltaCodec`'s own Javadoc): reusing `MyersDiff` buys correctness confidence (the algorithm is already exhaustively tested) since delta encoding is only attempted between objects already believed similar (same kind, chosen by the O(1) base heuristic), keeping d small in the common case ‚Äî at the cost of the O(n¬∑d) (worst case O(n¬≤)) bound applying per encode attempt instead of a guaranteed-linear one when that similarity assumption is wrong; `MAX_DELTA_DEPTH` is a separate, orthogonal mitigation for reconstruction cost, not for this. |
| **Packfile decode (delta)** | `pack/PackReader.java:read` (line 36), applying `pack/DeltaCodec.java:decode` (line 54) per delta entry | O(pack size) total: one forward pass, one inflate per entry, one `DeltaCodec.decode` per delta entry against an already-resolved base (see "Object read: packed" above for why this doesn't multiply by chain depth). `decode` itself is O(target size): a single pass over the delta's copy/insert instructions, each writing directly into the output array (`System.arraycopy`, lines 79/82) with no re-scanning. | O(pack size): all resolved objects held in the returned map (this is exactly `PackedObjectStore`'s documented tradeoff ‚Äî eager, not lazy). | Streaming decode that discards each object after use unless it is a delta base needed later. | Rejected for the same reason as packed reads above: simplicity over a real Git-style `.idx`-backed lazy resolver. |
| **Transfer negotiation (want/have)** | `cairn-transfer/UploadPackHandler.java:buildResponse` (line 70), reachability via `ObjectClosure.java:from` (line 29) | `ObjectClosure.from`: `RevWalk.history` over the starting commits (O(V log V + E), see "DAG walk" ‚Äî same tightened bound applies here, not the looser O((V+E) log V)), then one tree walk per commit via `:walkTree` (line 45) that adds each tree/blob id to a `LinkedHashSet` guarded by `out.add(treeId)` (line 46) ‚Äî an object already added (a shared subtree from an earlier commit) is skipped in O(1), so the tree-walk portion costs O(total *distinct* tree+blob objects reachable), not O(commits √ó tree size). `buildResponse` computes this closure for `wants` **and independently for `haves`**, then filters wants by "not in haveClosure" (O(1) average lookup per want) before calling `PackWriter.writePack` on just the missing set. **Why the ordered `history` walk is used here rather than the cheaper unordered `reachableFrom`:** this is not incidental ‚Äî `ObjectClosure.from`'s own comment (line 28) states the commits are walked oldest-first specifically "so a pack built from this set tends to place related blob revisions near each other," because the resulting `LinkedHashSet`'s iteration order becomes the `objectIds` list handed to `PackWriter.writePack`, and `PackWriter`'s delta-base heuristic (`lastOfKind`, "Packfile encode" row) picks its candidate base from whatever object of the same kind came immediately before it *in that same order*. An unordered `reachableFrom` walk would remove the V log V term but would scramble that ordering, degrading how often `PackWriter` finds a good delta base ‚Äî a real correctness-of-pack-quality tradeoff, not a free removable log factor. | O(size of each closure) ‚Äî both are materialized in full as `LinkedHashSet`s. | Incremental/cached closures reused across requests. | Not built: every request recomputes both closures from scratch. **This is worth stating precisely, since the architecture doc's phrasing ("the work... [is] proportional to the difference between client and server") is not quite what the code does**: the *bytes sent* are proportional to the missing set (`wantClosure \ haveClosure`), but the *work to compute reachability* is proportional to the full closures of both `wants` and `haves`, not to their difference ‚Äî a client with a huge amount of existing history still costs the server a full walk of that history's closure to confirm what it already has. |
| **Permission resolution** | `dev.cairn.api.permission.DefaultPermissionResolver.java:effectiveRole` (line 54), recursing via `:collectTeamGrants` (line 73) | O(D + T). To state D precisely rather than collapsing it to "O(1)": `effectiveRole` makes exactly two direct-grant-phase lookups (`lookup.ownerRole`, line 61; `lookup.directGrant`, line 62), each an O(1) point query against `GrantLookup` ‚Äî there is no loop over a variable-size collection of direct grants in this implementation (each returns a single `Optional<Role>`, not a list). So D here is a fixed constant (2 point lookups), each individually O(1); the correct way to read this is "iterating D direct-grant checks costs O(D) with each check O(1)," which in this code happens to collapse to O(1) only because D is hardcoded at 2, not because a direct-grant phase is inherently free ‚Äî a version of `GrantLookup` that returned multiple simultaneous direct grants per principal would make this phase genuinely O(D). T = edges walked in the nested team hierarchy via `collectTeamGrants`, the real variable term, bounded by both a hard depth cap (`MAX_TEAM_DEPTH = 50`, `DefaultPermissionResolver.java:45`) and a `visited` set (guards a genuinely cyclic team structure), so a pathological or cyclic hierarchy cannot turn one permission check into an unbounded traversal. | O(T) for the visited set and the recursion depth (bounded by the depth cap regardless of actual hierarchy size). | Cache the resolved role per (principal, repo), O(1) amortized. | Rejected as the v1 default (security doc, section 3.4): recomputing on every call is simple and always correct; caching needs an invalidation story (any grant, membership, or ownership change must invalidate the affected entries) that is not built. Recompute-every-time is the deliberate tradeoff. |
| **Trigram index build** | `cairn-vcs/search/TrigramIndex.java:add` (line 37); per-repo orchestration in `dev.cairn.api.search.RepoSearchIndexService.java:walk`/`buildIndex` (lines 90/98) | `add`: O(content length) ‚Äî one O(1) 3-character `substring` per position (`:trigramsOf`, line 107), each inserted into a `HashSet` (dedup within the document) then the postings map. Building a whole repo's index (`buildIndex`) calls `add` once per file, walking the tree exactly like `ObjectClosure` (recursive, `:walk`, line 98), so total build cost is O(total indexed content length), bounded by the per-file (`MAX_FILE_BYTES = 512,000`, line 44) and per-repo (`MAX_FILES = 20,000`, line 45) caps named explicitly rather than silently truncating. | O(total indexed content length): every document's full text is retained (`contentByPath`) alongside the postings map, so both the index and a full copy of every indexed file's content live in memory for the lifetime of that repo's index. | Persist the index to disk / an external search service. | Not built: in-memory only, rebuilt from scratch (`RepoSearchIndexService`, a new build discards the old index only once the new one finishes) whenever the branch tip moves, off the request thread. The real cost this defers is index staleness under concurrent pushes and process restarts, named as a gap rather than solved. |
| **Trigram query** | `cairn-vcs/search/TrigramIndex.java:search` (line 61), candidate narrowing at `:candidateDocuments` (line 80), verification at `:matchingLines` (line 96) | O(query length) to extract the query's own trigrams (at least 3, or the query is rejected as too short before this point). Candidate narrowing: for each query trigram, one O(1) posting-list lookup, then a `HashSet` intersection ‚Äî cost is O(sum of the posting-list sizes touched), shrinking as more trigrams are intersected (the loop returns early once the running intersection is empty, line 89-91). Verification: for each surviving candidate document, one O(document length) scan (line-by-line `contains` check) ‚Äî necessary because trigram membership alone does not prove the query substring occurs *contiguously and in order* (proven directly by `TrigramIndexTest.trigramFalsePositivesAreRejectedByCandidateVerification`, which constructs a document containing every trigram of a query without containing the query itself). | O(number of surviving candidates) for the intermediate set. | Return matches on trigram membership alone (skip verification). | Rejected: would produce false positives, proven by the test above. The verification pass is real, load-bearing work, not a formality ‚Äî its cost (O(candidates √ó their length)) is the actual price of trigram indexing's speedup over a naive O(total code) grep, not something to elide. |
| **Blame** | `cairn-vcs/blame/Blame.java:blame` (line 57) | O(commits walked along the first-parent chain √ó diff cost per step): at each step, one `MyersDiff.diff` call (`:blame`'s loop body) between the parent's and the current commit's version of the file, O(N¬∑D) per the "Diff" row above, N = file's line count at that step. The walk stops as soon as every target line is attributed (the `while (!currentLines.isEmpty())` guard) rather than always consuming the whole history, so a file whose lines are all recently touched is cheap regardless of total repository history length. Root-commit and file-not-yet-added cases are handled without extra passes (`:fileLinesAt`, line 111, returns empty rather than throwing). | O(target file's line count) for the attribution array and the per-step surviving-lines list (shrinks every step an `INSERT` attributes lines away). | Follow every parent at a merge (real Git's `--rebase-merges`-style full history); detect renames across files. | Rejected (documented in `Blame`'s own Javadoc): only the first-parent chain is walked, matching this project's existing merge/rename scope boundary (`TreeMerger`'s own documented limitation) rather than adding move/copy detection or multi-parent traversal machinery beyond what blame needs to prove real line-level provenance. |

## Notes on scope

`Repository.status()` (`cairn-vcs/repository/Repository.java:status`) is intentionally
not in this table: its own Javadoc already states the cost precisely (O(tracked file
bytes + working tree entries), since the index carries no stat cache to short-circuit
unchanged files), and it is a working-directory operation with no server-side
counterpart to reason about at scale (see `SUMMARY.md`'s FR audit).

`Rebase` (`cairn-vcs/merge/Rebase.java:rebase`) is a repeated application of the
three-way-merge and diff costs above (one `TreeMerger`-equivalent three-way merge per
replayed commit) plus the ancestor-walk cost to find the merge base and the
first-parent chain to replay; its own Javadoc already states this and the same
first-parent-only limitation as `Blame`, so it is not duplicated here as a separate
row.
